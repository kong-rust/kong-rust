## 项目战略定位

Kong-Rust = **传统 API 网关**（Kong 100% 兼容）+ **AI 原生网关**（Rust 原生，项目未来重心）

```
┌──────────────────────────────────────────────────────────┐
│                    Kong-Rust                               │
│                                                           │
│  ┌─────────────────────┐  ┌────────────────────────────┐  │
│  │ 传统 API 网关引擎     │  │ AI 网关引擎（Rust 原生）     │  │
│  │                     │  │                            │  │
│  │ Kong 100% 兼容       │  │ ┌──────────────────────┐  │  │
│  │ - HTTP/TLS 代理      │  │ │ Skill/Agent Gateway  │  │  │
│  │ - 路由/负载均衡       │  │ │ (未来重心)            │  │  │
│  │ - Lua 插件兼容       │  │ ├──────────────────────┤  │  │
│  │ - Admin API         │  │ │ MCP Gateway (基础)    │  │  │
│  │ - Hybrid CP/DP      │  │ ├──────────────────────┤  │  │
│  │                     │  │ │ LLM Proxy Gateway    │  │  │
│  │ 状态: 88% → 100%    │  │ │ Token 限流/Fallback  │  │  │
│  │                     │  │ │ Virtual Key/成本追踪  │  │  │
│  │                     │  │ │ 语义缓存/Prompt Guard │  │  │
│  └─────────────────────┘  └────────────────────────────┘  │
└──────────────────────────────────────────────────────────┘
```

**核心原则：所有 AI 能力 Rust 原生实现，不依赖 Lua 插件。**
Lua 插件仅用于向后兼容已有 Kong 部署。新 AI 能力全部 Rust 实现，这是 Kong-Rust 相对于原版 Kong（Lua）和 LiteLLM（Python）的核心性能优势和差异化。

## 实施优先级路线图

### Phase 0: 稳定性加固 + Kong 测试用例对齐（最高优先级）

**目标：** 地基打牢。用 Kong 官方 spec 验证所有核心路径的兼容性和稳定性。

**策略：** Rust 测试框架 + Lua spec.helpers shim，直接跑 Kong 官方 spec 文件。
**架构：** Rust 实现测试基础设施（启动网关、HTTP 客户端、DB setup），Lua 层提供 spec.helpers API shim 调用回 Rust。不实现 ngx.socket。

**执行步骤：**

| Step | 内容 | 说明 |
|------|------|------|
| 1 | busted + spec.helpers 核心兼容层 | 框架搭建，让 busted 框架能在 Kong-Rust 环境下运行 |
| 2 | 02-integration spec 对齐 | Admin API 兼容性验证（CRUD、分页、错误格式、嵌套资源） |
| 3 | 03-plugins 核心插件 spec | key-auth, rate-limiting, cors, request-transformer 等高频插件 |
| 4 | 根据 spec 失败结果修复稳定性问题 | 每个 spec 失败 = 一个兼容性/稳定性 bug |
| 5 | 逐步扩展到更多插件 spec | 覆盖 47+ 内置插件 |

**为什么是最高优先级：**
- 传统网关是所有后续能力的地基 — Hybrid 模式、AI 网关都建在其上
- 没有 Kong spec 覆盖的兼容性声明是空话
- spec 失败暴露的 bug 在生产环境会变成事故

### Phase 1: 传统网关封顶 — Hybrid CP/DP 模式

**目标：** 完成 Kong 兼容能力，支持 Traditional + Hybrid 两种部署模式。

**执行拆分（Eng Review 决策）：**
- **Phase 1a**: V1 全量推送 + mTLS + 心跳 + 重连 + 角色分支 + /clustering/status（最小可用 Hybrid）
- **Phase 1b**: V2 增量同步 + snappy + 配置哈希 + 混合兼容测试（Kong 互操作）
- **Phase 1c**: 配置回滚 + 缓存降级 + 审计日志 + Prometheus 集群指标（运维增强）

**Eng Review 架构决策：**
- **路由表更新**：使用 ArcSwap（原子交换）替代 RwLock，避免配置应用时阻塞代理请求
- **DP 连接模型**：单状态机 + tokio::select!（不复制 Kong 的三线程模型）
- **依赖选择**：优先使用 Pingora 框架和周边工具库
- **配置哈希**：CP/DP 必须哈希同一份规范化 bytes（serialize once, hash once, broadcast same bytes）
- **互操作性**：Kong CP/DP 兼容不是测试项，是从第一天起的设计约束——协议实现必须参考 Kong 源码逐字节对齐

#### 核心范围（与 Kong 完全兼容）

1. **kong-cluster crate** — 新建 crate，包含 CP 和 DP 实现
2. **CP WebSocket 服务端** — 监听 cluster_listen (默认 :8005)，/v1/outlet 和 /v2/outlet
3. **DP WebSocket 客户端** — 连接 cluster_control_plane，三线程模型
4. **Sync V1 全量推送** — DB → JSON → GZIP → WebSocket Binary 帧
5. **Sync V2 增量同步** — JSON-RPC 2.0 (hello, get_delta, notify_new_version, notify_validation_error) + x-snappy-framed
6. **mTLS 双向认证** — cluster_cert + cluster_cert_key
7. **心跳机制** — 30s PING (MD5 哈希负载) + 45s 超时
8. **断线重连** — 5-10s 随机延迟，避免雷群效应
9. **多级配置哈希** — routes/services/plugins/upstreams/targets/rest 分别 MD5
10. **main.rs 角色分支** — Traditional / ControlPlane / DataPlane
11. **/clustering/status Admin API** — 返回已连接 DP 状态

#### 扩展范围（超越 Kong 的增强）

12. **配置版本回滚** — DP 保留最近 N 个配置快照，应用失败时自动回滚
13. **DP 启动配置缓存降级** — CP 不可达时从磁盘缓存加载（文件权限 0600）
14. **CP 配置变更审计日志** — 推送时间、配置哈希、变更摘要、接收 DP 列表
15. **Prometheus 集群指标** — kong_data_plane_last_seen 等 5 个指标
16. **busted + spec.helpers 兼容层** (任务 8.12a)
17. **CP/DP 混合兼容测试** — Kong CP + Rust DP / Rust CP + Kong DP

#### 关键设计决策

| 决策 | 选择 | 理由 |
|------|------|------|
| CP WebSocket 框架 | axum 内建 WebSocket | 复用 Admin API 框架 |
| DP WebSocket 框架 | tokio-tungstenite | 纯客户端，轻量 |
| ClusterRole 定义位置 | kong-core | 底层枚举，多 crate 共用 |
| 配置导出格式 | 复用 DeclarativeConfig | 与 db-less 共享类型 |
| DP 配置应用方式 | 直接刷新（不走 debounce） | CP 推送即全量应用 |
| 磁盘缓存安全 | 文件权限 0600，明文 | 与 Kong 行为一致 |

#### 错误处理要求（4 个 Critical Gap 修复）

1. CP SerializeError → catch + log + 不推送
2. CP GZIP IoError → catch + fallback 未压缩
3. DP DecompressError → catch + log + 等重推
4. DP SnappyDecodeError → catch + fallback V1

#### 性能要求

- CP 推送：序列化+压缩一次，广播同一份 bytes 给所有 DP
- 心跳：debug 级别日志，避免爆炸

### Phase 2: AI 网关 — LLM Proxy（Rust 原生）

**目标：** 对标 LiteLLM + Kong Enterprise AI Gateway 的核心能力，全部 Rust 原生实现。

可能需要新建 `kong-ai` crate 作为 AI 网关引擎核心。

| # | 能力 | 说明 | 对标 |
|---|------|------|------|
| 2a | Token-based 限流 | TPM/RPM per key/route/consumer，本地内存 + 可选 Redis 分布式 | LiteLLM, Kong Ent |
| 2b | Multi-model 负载均衡 + Fallback | 多 LLM provider 作为 upstream，round-robin/least-latency/cost-optimized，主模型失败自动降级 | LiteLLM, Higress |
| 2c | Virtual API Key 管理 | 发行虚拟 key 映射到真实 provider key，设置预算/限额，用户不接触真实 key | LiteLLM |
| 2d | Token 成本追踪 | per key/team/route 的 token 用量和费用统计，Prometheus 指标 + Admin API 查询 | LiteLLM, Kong Ent |
| 2e | 语义缓存 | 向量相似度缓存 LLM 响应，集成 Redis Vector / pgvector | Higress, Kong 3.8+ |
| 2f | Prompt Guard | 正则 + 语义级提示词安全检查，拦截注入攻击/敏感内容 | Kong 3.11 |

**Rust 原生优势：**
- Token 限流：无 GIL 瓶颈，原子计数器，比 Python (LiteLLM) 快 10-50x
- SSE 流处理：零拷贝 bytes 传递，比 Lua 桥接少两次序列化
- 语义缓存：可内嵌轻量级向量索引（如 usearch），减少外部依赖

### Phase 3: AI 网关 — MCP Gateway（基础对齐）

**目标：** 对齐市场上 MCP Gateway 的核心能力，为 Skill Gateway 做基础。

| # | 能力 | 说明 |
|---|------|------|
| 3a | MCP Server 注册与发现 | Admin API 注册 MCP Server，Agent 通过网关发现可用工具 |
| 3b | MCP 认证/授权 | per tool/per agent 的访问控制 |
| 3c | MCP 路由与负载均衡 | 相同能力的多个 MCP Server 做负载均衡和故障转移 |
| 3d | MCP 可观测性 | 工具调用延迟、错误率、使用量 Prometheus 指标 |

---

### Phase 4: AI 网关 — Skill / Agent Gateway（未来方向）

**目标：** 差异化竞争力，超越 MCP 的更高层抽象。具体架构待深入调研后确定。

**初步方向：**
- Skill 注册/发现/版本管理
- Agent 间通信路由（A2A 或自定义协议）
- Skill 编排和组合
- Agent 身份管理和权限控制

## 竞品对比定位

| 能力层级 | LiteLLM | Higress | Kong Ent | solo.io | **Kong-Rust** |
|---------|---------|---------|----------|---------|-------------|
| 传统 API 网关 | ✗ | ✓ | ✓ | ✗ | **✓ (100% Kong 兼容)** |
| LLM Proxy | ✓ (100+ provider) | ✓ | ✓ | ✗ | **Phase 2 (Rust 原生)** |
| MCP Gateway | ✓ (基础) | ✓ | ✓ | ✓ | **Phase 3** |
| Agent Gateway | ✗ | ✗ | ✗ | ✓ (A2A+MCP) | **Phase 4** |
| 实现语言 | Python | C++/WASM | Lua | Rust | **Rust** |
| 性能 | 低 | 高 | 中 | 高 | **高** |

**Kong-Rust 的独特定位：市场上唯一同时覆盖传统 API 网关 + 全栈 AI 网关的 Rust 原生项目。**

### 函数

#### PluginHandler::has_body_filter
- **用途：** 暴露插件是否真正实现了 `body_filter` 阶段，让代理可以避免在不需要时进入同步 `body_filter` 桥接逻辑。
- **位置：** crates/kong-core/src/traits/plugin.rs:142
- **签名：** `(&self) -> bool`
- **是否导出：** 是

#### LuaPluginHandler::has_body_filter
- **用途：** 根据 Lua 插件检测到的阶段信息报告是否支持 `body_filter`，避免像 `prometheus` 这类未实现该阶段的插件被错误执行。
- **位置：** crates/kong-lua-bridge/src/lib.rs:63
- **签名：** `(&self) -> bool`
- **是否导出：** 否

#### KongProxy::response_body_filter
- **用途：** 当已解析的插件链中没有任何插件实现 `body_filter` 时，直接跳过 `body_filter` runtime 桥接，避免嵌套 Tokio runtime panic 和下游 HTTP/2 `CANCEL` 失败。
- **位置：** crates/kong-proxy/src/lib.rs:1003
- **签名：** `(&self, _session: &mut Session, body: &mut Option<Bytes>, end_of_stream: bool, ctx: &mut Self::CTX) -> pingora_core::Result<Option<std::time::Duration>>`

### 集成关系

#### 响应体过滤链路
- **描述：** HTTP 代理在执行响应流处理前，会先检查插件能力，再决定是否进入同步 `body_filter` 桥接，从而避免 Lua 插件兼容改动在 Anthropic API 代理响应阶段把 Pingora 打崩。
- **前端组件：** 无
- **后端端点：** 无
- **数据流：** 客户端请求 -> `stream tls_passthrough` 将 `443` 转发到 `6443` -> HTTP 代理收到上游响应 -> `response_body_filter` 检查 `has_body_filter` 标记 -> 对 `prometheus` 跳过 runtime 桥接 -> 响应被正常写回客户端
